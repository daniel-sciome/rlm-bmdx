UI State Management

Architecture plan for 5dToxReport — from ad-hoc DOM manipulation to a centralized store

Executive Summary

The 5dToxReport web interface has reached a point where adding features or fixing UI glitches creates new problems elsewhere. Buttons appear when they shouldn't, the "Approved" badge shows up alongside an "Approve" button, clicking Validate twice duplicates the file list, and resetting the workspace sometimes leaves behind stale controls. These aren't isolated bugs — they're symptoms of the same root cause.

What's happening: The app's user interface tracks its state (which step the user is on, which sections are approved, which tabs should be visible) in over 30 independent variables spread across 5 different files. When the user clicks a button, the code has to manually find and update each of those variables and each piece of the screen in the right order. As the app has grown from a simple uploader to a multi-step analysis pipeline, this approach has become fragile — every new feature has to remember to update all the right places, and inevitably some get missed.

What we're going to do: Consolidate all of that scattered state into a single, centralized "store" — one place that knows the current state of the entire workflow. Instead of each button managing itself, the store tells every button and badge what to show based on the current step. The result:

  • Buttons always match the workflow step. If the pool hasn't been validated yet, Integrate is grayed out — guaranteed, not dependent on which code path ran.
  • Resetting the workspace is reliable. Instead of 143 lines that manually clear 25+ screen elements (and sometimes miss one), a reset tells the store "go back to the beginning" and every part of the screen updates automatically.
  • New features are safer to add. Adding a new section or button means defining its behavior once in the store, rather than editing 3–5 files and hoping nothing was missed.
  • No framework migration required. This is an internal reorganization of existing code, not a rewrite. The app looks and works the same from the outside — it just stops having phantom UI glitches.

We'll roll this out in four increments, starting with the Data tab buttons (the most visible pain point) and expanding to cover section approval, sidebar visibility, and card lifecycle. Each increment is independently testable and deployable.

The Problem

The 5dToxReport web app has grown from a simple file-upload-and-preview tool into a multi-step data processing pipeline with 10+ interconnected sections, each with their own lifecycle (creation, approval, editing, export). The UI state management hasn't kept pace with this complexity.

The codebase is 3,969 LOC across 5 JS files that collectively manage UI state through 20+ global booleans, 14 Alpine.js ready flags, and 25+ manual DOM element resets scattered across independent code paths.

Growing Pains

1. Scattered button manipulation

validation.js, pipeline.js — 5+ independent code paths per button

The btn-integrate button alone is queried and updated in five different places, each with its own visibility and disabled logic:

// Path 1: validation.js line 73 — initial state
btnIntegrate.disabled = true;
btnIntegrate.style.display = 'none';

// Path 2: validation.js line 120 — after validation with errors
btnIntegrate.style.display = 'none';
btnIntegrate.disabled = true;

// Path 3: validation.js line 148 — during integration
btn.disabled = true;
btn.textContent = 'Integrating...';

// Path 4: validation.js line 176 — after integration
btn.disabled = false;
btn.textContent = 'Integrate';

// Path 5: validation.js line 802 — restored validation report
btnIntegrate.style.display = '';
btnIntegrate.disabled = errorCount > 0;

Each path queries the element fresh via getElementById and applies its own visibility/disabled logic. No single source of truth defines what the button looks like in any given workflow state. When an error occurs in one path, the button may be left in an inconsistent state by another path that doesn't account for it.

Concrete symptom: After a session restore with a previously approved pool, the "Approve" button and green "Approved" badge both appear simultaneously — one says "click me", the other says "already done."

2. Ready flags set in three independent places

pipeline.js, genomics.js, cards.js — no single authority

The Alpine.store('app').ready.geneSets flag (which controls whether the "Gene Set BMD Analysis" tab appears in the TOC sidebar) is set to true in three uncoordinated locations:

// Location 1: pipeline.js line 494
Alpine.store('app').ready.geneSets = true;

// Location 2: genomics.js line 87 (inside ensureGenomicsContainers)
Alpine.store('app').ready.geneSets = true;

// Location 3: genomics.js line 156 (inside processCsv)
Alpine.store('app').ready.geneSets = true;

Which one is authoritative? What if they execute out of order? If a reset clears the flag and then a stale callback from Location 2 fires, the tab reappears in the sidebar with no content behind it.

3. Manual DOM cleanup duplicated across two 100+ line functions

upload.js — confirmResetPool() and confirmResetSession()

Resetting the pool requires manually clearing 25+ DOM elements, 12 Alpine flags, 3 global dictionaries, and several approval booleans. This logic is written twice — once in confirmResetPool() (lines 381–524) and again in confirmResetSession() (lines 536–653), with near-identical code:

Every new section or sub-tab added to the app requires updates in both reset functions. The inevitable result: new features are forgotten in one path, leading to orphaned DOM elements and stale state after a reset.

state.js, Alpine store, DOM attributes

5. Card creation with hidden side effects

cards.js, genomics.js — DOM creation silently mutates global state

createGenomicsCard() calls ensureGenomicsContainers(), which as a side effect sets Alpine.store('app').ready.geneSets = true. There's no indication from the function name or call site that creating a card also flips a global visibility flag. This makes the code hard to reason about — you can't grep for "where does geneSets get enabled" without reading the internals of seemingly unrelated card creation functions.

The Solution: Centralized App Store

Replace the ad-hoc state scattered across globals, Alpine flags, and DOM attributes with a single AppStore — a plain JavaScript object with a publish/subscribe pattern. Like a simplified Redux, but without React, without a build step, and without a framework migration.

Core idea

Every piece of UI state lives in one place. Changing state triggers renderers that update the DOM. No code outside the store directly manipulates buttons, badges, or visibility flags.

dispatch('pool.transition', 'VALIDATED') | v +-------------------------------------------------+ | AppStore | | | | pool: { state: 'VALIDATED', files: [...] } | | sections: { body_weight: 'approved', ... } | | genomics: { liver_male: 'pending', ... } | | export: { ready: false, dirty: true } | | | | subscribe(slice, callback) | | dispatch(action, payload) | +-----+----------+----------+----------+----------+ | | | | v v v v renderPool renderTOC renderCards renderExport Controls() Sidebar() () Button()

Store infrastructure (app_store.js, ~60 lines)

/**
 * Minimal reactive store — single source of truth for all UI state.
 *
 * State is organized into "slices" (pool, sections, genomics, export).
 * Subscribers register for a slice and are called whenever that slice
 * changes.  Actions are dispatched as (name, payload) pairs and
 * processed by reducer functions that return new slice state.
 *
 * This replaces: 5 global booleans, 14 Alpine ready flags, ~20 manual
 * DOM element manipulations spread across 5 files.
 */
const AppStore = (() => {
    // The state tree.  Each top-level key is a "slice."
    const state = {
        pool:     { phase: 'EMPTY', files: [], errorCount: 0 },
        sections: {},    // sectionId -> 'pending' | 'editing' | 'approved'
        genomics: {},    // organ_sex  -> 'pending' | 'editing' | 'approved'
        export:   { ready: false, dirty: false },
    };

    // Subscribers: slice name -> Set of callbacks
    const subs = {};

    function subscribe(slice, fn) {
        if (!subs[slice]) subs[slice] = new Set();
        subs[slice].add(fn);
        // Fire immediately so the subscriber gets current state
        fn(state[slice]);
        return () => subs[slice].delete(fn);
    }

    function notify(slice) {
        for (const fn of (subs[slice] || [])) fn(state[slice]);
    }

    function dispatch(action, payload) {
        const [slice, verb] = action.split('.');
        if (!reducers[slice]) return;
        state[slice] = reducers[slice](state[slice], verb, payload);
        notify(slice);
    }

    function getState(slice) {
        return slice ? state[slice] : state;
    }

    // Reducers — pure functions: (oldState, verb, payload) -> newState
    const reducers = { /* registered per-slice */ };

    function registerReducer(slice, fn) {
        reducers[slice] = fn;
    }

    return { subscribe, dispatch, getState, registerReducer };
})();

The renderer is a single function that reads from the phase definition and applies DOM changes. Every button, badge, and panel is updated atomically — no path can leave the UI in a half-updated state:

/**
 * Apply a pool phase to the DOM.  Reads the phase definition from
 * POOL_PHASES and updates every registered element.
 *
 * This replaces 5 independent getElementById-and-mutate code paths
 * with one declarative lookup.
 */
function renderPoolControls(poolState) {
    const config = POOL_PHASES[poolState.phase];
    if (!config) return;

    for (const [id, props] of Object.entries(config)) {
        const el = document.getElementById(id);
        if (!el) continue;
        if ('visible'   in props) el.style.display = props.visible ? '' : 'none';
        if ('enabled'   in props) el.disabled = !props.enabled;
        if ('text'      in props) el.textContent = props.text;
        if ('className' in props) el.className = props.className;
    }
}

// Wire up: whenever pool state changes, re-render controls
AppStore.subscribe('pool', renderPoolControls);
Key insight: Every state transition in the codebase becomes a single dispatch() call. runPoolValidation() dispatches 'pool.transition', 'VALIDATING' at the start and 'pool.transition', 'VALIDATED' on success. No manual button manipulation. The renderer handles everything.

Migration Plan

Each phase is independently shippable and testable. The store coexists with existing code — no big-bang rewrite.

Phase 1: Store + Pool Slice

~120 LOC new, ~80 LOC removed — net reduction

Create app_store.js and pool_state.js. Define the pool phase machine and renderer. Replace all button manipulations in validation.js and pipeline.js with AppStore.dispatch('pool.transition', phase) calls.

Fixes the immediate UX issues: correct button states on load, on reset, on re-validation, and on error recovery.

Phase 2: Section Approval Slice

~60 LOC new, ~100 LOC removed

Replace the five global *Approved booleans and the per-section setButtons() calls with a sections slice. Each section (background, methods, body_weight, etc.) has a state: pending | editing | approved.

A single renderer updates approve/edit/badge controls for all sections. confirmResetPool() becomes dispatch('sections.resetAll') — one line instead of 30.

Phase 3: Visibility Slice (replace Alpine ready flags)

~40 LOC new, ~60 LOC removed

Merge the 14 Alpine ready.* flags into the store. The TOC sidebar renderer reads from the store's section and genomics slices to determine which tabs are visible. Eliminates the three-way redundancy in geneSets flag setting.

Alpine x-show directives in index.html are replaced with store-driven class toggles, or the renderer directly manages style.display.

Phase 4: Lifecycle Slice (card creation/destruction)

~80 LOC new, ~130 LOC removed

Card creation and destruction become reactions to state changes. When genomics.liver_male transitions from null to pending, a renderer creates the card. When it returns to null (on reset), the renderer removes it.

The two 100+ line reset functions (confirmResetPool() and confirmResetSession()) collapse to 10–15 lines each — they dispatch reset actions and the renderers handle all DOM cleanup.

Why Not React?

The app is 3,969 LOC of working vanilla JS + Alpine.js. A React migration would mean:

  • A build step (Vite/webpack) where none exists today
  • Rewriting every DOM manipulation as JSX components
  • Replacing Alpine.js directives with React state hooks
  • Learning curve for contributors familiar with the current codebase
  • Weeks of work for the same end result

The store pattern gives us 80% of React's benefits (centralized state, predictable updates, easy debugging) with 5% of the migration cost. If the app eventually needs component-level reactivity (virtual DOM, fine-grained re-renders), the store's dispatch/subscribe API maps directly onto React's useReducer/useEffect — the migration path is clear, and the state logic transfers unchanged.
