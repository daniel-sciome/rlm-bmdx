1. The Domain: Computational Toxicology Report Generation

The National Institute of Environmental Health Sciences (NIEHS) publishes standardized reports characterizing the biological potency of chemicals through 5-day genomic dose-response studies. These reports follow a rigid template: animal condition data (body weights, organ weights), clinical pathology endpoints (clinical chemistry, hematology, hormones), internal dose assessment (tissue concentrations), and transcriptomic benchmark dose analysis — all organized into numbered tables, narrative sections, and statistical summaries that must match an established reference format.

Generating one of these reports is labor-intensive. The data arrives in heterogeneous formats: BMDExpress 3 binary .bm2 files containing dose-response model results, NTP-format CSV and TXT files with per-animal measurements, Excel workbooks with study rosters, and gene expression matrices with thousands of probes. Each file type carries different metadata, uses different conventions, and represents a different tier in a hierarchy of data provenance:

  • Tier 1 (.bm2) — BMDExpress 3 modeling results with benchmark dose estimates, model fit statistics, and NIEHS classification (viable, NVM, NR, UREP)
  • Tier 2 (.txt/.csv) — Pivoted dose-response tables with mean ± SE per dose group, derived from raw animal data
  • Tier 3 (.xlsx) — Raw study files with individual animal measurements, dose assignments, and selection metadata

The rlm-bmdx application automates this report generation pipeline: upload heterogeneous study files, validate and cross-reference them, merge into a unified data structure, run NTP-mandated statistical tests (Jonckheere trend, Williams/Dunnett pairwise), compute benchmark doses via the EPA’s recommended model selection algorithm, generate narrative prose, and export the final report as a tagged PDF matching the NIEHS reference layout.

The Statistical Pipeline

Two independent statistical systems operate on the integrated data, and their independence is a critical business rule:

System Purpose Implementation Output
NTP Statistical Gate Determine if an endpoint shows a significant dose-response trend Jonckheere trend test (Python) + Williams/Dunnett pairwise (Java via sciome-commons-math) responsive flag, significance markers (* / **), LOEL/NOEL
BMD Modeling Estimate the benchmark dose for each endpoint BMDExpress 3 (.bm2 prefilter) + pybmds (15 continuous models, EPA algorithm) BMD, BMDL, BMDU, model name, NIEHS classification

An endpoint can be NTP-significant without having a viable BMD (model failed), and it can have a viable BMD without passing the NTP gate (the statistical trend test didn't reach significance). Both results appear in the tables — the BMD column reflects modeling status, not the NTP gate.

2. System Architecture and Sources of Truth

The system is a FastAPI web application (Python 3.12) with a JavaScript frontend (Alpine.js). It is built around three canonical sources of truth, each enforced by architectural rules documented in the project’s CLAUDE.md:

Source 1: The Integrated Dataset

integrated.json (BMDProject format) is the merged output of all uploaded files, produced by the integrate_pool() function which calls BMDExpress 3’s Java-based IntegrateProject tool. Supplemented by *.sidecar.json files that preserve per-animal metadata the pivot step discards (observation day, selection group, terminal flag). Rule: No table builder or report generator reads from individual file pool files.

Source 2: The Document Structure Tree

document_tree.py defines the complete NIEHS report structure as a tree of DocNode objects. Heading hierarchy, section ordering, table numbering, TOC navigation, and PDF preview filtering all derive from this tree. Rule: No part of the system hardcodes document structure.

Source 3: The AppStore State Machine

web/js/app_store.js centralizes all UI state. The pool workflow progresses through a linear state machine:

EMPTY → UPLOADED → VALIDATED → INTEGRATED → APPROVED

Phase is derived from artifact state (what files and reports exist on disk), never set imperatively. The function derivePoolPhase() examines artifacts and returns the correct phase.

3. The Domain Model Refactor

On March 15, 2026, a major refactor was committed across five sequential commits that replaced the system’s entire file classification scheme. Previously, files were identified by a single monolithic domain string (e.g., "body_weight_inferred", "clinical_chemistry_tox_study"). This was brittle — the same string encoded three orthogonal concerns, making it impossible to distinguish a raw animal data file from a BMDExpress modeling result for the same measurement platform without parsing the string.

The New Taxonomy

AxisValuesSemantics
Provider Apical, Observations Top-level grouping: continuous dose-response vs. categorical event data
Platform Body Weight, Organ Weight, Clinical Chemistry, Hematology, Hormones, Tissue Concentration, Clinical The measurement domain — what biological system is being assessed
dataType tox_study, inferred, gene_expression The provenance tier — raw animal data, BMDExpress model results, or transcriptomic probes

What Changed

  • FileFingerprint.domain → .platform + .data_type
  • Coverage matrix keys: "Body Weight" → "Body Weight|tox_study" (compound keys)
  • Java ExperimentDescription.domain → .dataType; providers "Study"/"Clinical Endpoint" → "Apical"/"Observations"
  • New metadata confirmation gate: user must confirm platform + dataType per file before integration
  • Long-format CSV conversion moved to upload time (originals preserved in _originals/)
  • UI: new PLATFORM_LABELS, coverage matrix collapses compound keys
Critical gap: The refactor was committed on March 15 but had received zero end-to-end validation by March 30 — a 15-day window during which 30+ additional commits were built on top of it. Manual testing through the browser was the only validation available, and it had not been performed.

4. Why Automated Testing Is Necessary

4.1 The Manual Testing Bottleneck

Prior to this test suite, the only way to verify the application worked was to start the server, open a browser, upload files, click through each workflow step, and visually inspect the results. A single manual walkthrough of the 9-point validation checklist took 30–60 minutes (not counting the ~10-minute processing time for BMD modeling). When something broke — which it frequently did after refactors — diagnosing the failure often consumed an entire conversation session (1–2 hours) of iterative debugging.

This is unsustainable for a system with 87 commits in 30 days and 7,500+ lines of Python spanning 10 key modules, 6 Java helper tools, and 5 external API integrations.

4.2 The Regulatory Context

The reports this system generates are used in regulatory toxicology — they inform decisions about chemical safety, exposure limits, and public health policy. Errors in the statistical pipeline (wrong Jonckheere p-value, misclassified BMD status, inflated N counts from including Biosampling Animals in Core Animal statistics) produce incorrect conclusions in the final report. A test suite that validates these computations against known-good outputs is not a luxury; it is a prerequisite for using this tool in a professional setting.

4.3 The Domain Model Refactor Risk

The provider/platform/dataType refactor touched every layer of the system:

  • Python data structures (FileFingerprint, coverage matrix, experiment descriptions)
  • Java interop (BMDExpress ExperimentDescription field names)
  • File I/O (metadata headers prepended to CSV/TXT, filename-based platform detection)
  • UI rendering (coverage matrix display, platform labels, metadata confirmation modal)
  • Session persistence (fingerprint format, validation report structure)

Any of these touchpoints could silently break while appearing to work in the others. The old .domain field might still appear in a serialized validation report; an experiment might be assigned to the wrong platform; the coverage matrix might use old-format keys that the UI doesn’t recognize. Without automated assertions, these regressions can only be discovered by manually exercising each path.

4.4 The Shared Mutable State Problem

The server maintains five module-level dictionaries that accumulate state across HTTP requests: 

_pool_fingerprints: dict[str, dict[str, FileFingerprint]]  # dtxsid -> file_id -> fingerprint
_integrated_pool:   dict[str, dict]                         # dtxsid -> merged BMDProject
_data_uploads:      dict[str, dict]                         # file_id -> {filename, path, type}
_bm2_uploads:       dict[str, dict]                         # bm2_id -> {filename, path, table_data, ...}
_csv_uploads:       dict[str, dict]                         # csv_id -> {filename, path, df}

Without explicit state isolation, tests that exercise these dicts in sequence will see each other’s side effects. The conftest.py fixture strategy — clearing all five dicts and patching SESSIONS_DIR across six modules — was essential to make tests independent.

4.5 The External Dependency Wall

The pipeline calls six Java CLI tools, the Claude API, five HTTP APIs, an LMDB database, and a CPU-bound scipy modeling library that takes 8+ minutes per full run. Any test that accidentally invokes these dependencies is either slow, flaky, or requires network access and API keys. The mock architecture must be comprehensive:

Mocked AtWhat It BlocksWhy

Design decision: Mocks are applied at the import site (e.g., pool_orchestrator.integrate_pool) rather than globally patching subprocess.run. This makes each test’s external dependencies explicit and ensures new Java tools don’t silently escape the mock boundary.

5.2 Fixture Strategy

Every test receives isolated state through four layers:

  1. Fresh tmp_path (pytest built-in) — unique temp directory per test, auto-cleaned
  2. Patched SESSIONS_DIR — redirected in session_store and every module that imports it (6 total: session_store, session_routes, llm_routes, background_server, style_learning, export_routes)
  3. Cleared shared state — all 5 module-level dicts emptied between tests
  4. Golden session data — real study data copied into tmp_path via shutil.copytree
SESSIONS_DIR pitfall: session_store.SESSIONS_DIR is a module-level Path computed at import time from an environment variable. Setting SESSIONS_DIR via monkeypatch.setenv alone is insufficient because the module has already been imported. The conftest must use monkeypatch.setattr on the module attribute directly, and must do so in every module that imported it by name.

5.3 Test Layers

LayerWhatI/OMockingTarget Time

INTEGRATION FastAPI TestClient with endpoint calls Disk (tmp_path) Java, pybmds, HTTP, LLM <2s

5.4 What Is NOT Mocked

A critical design decision: several bmdx_pipe functions are not mocked because they are pure Python and safe to exercise with real data:

  • fingerprint_file() — reads file headers, detects platform/dataType from column names and metadata headers. This is the primary assertion target for the domain model refactor.
  • validate_pool() — cross-checks fingerprints for dose consistency, sex coverage, redundancy. Pure Python.
  • lightweight_validate() — quick validation of a new file against existing pool. Pure Python.
  • build_clinical_obs_tables() — parses categorical CSV data into incidence rows. Pure Python.

Leaving fingerprint_file() unmocked is what makes test_pool_validate.py the highest-value test: it exercises the real fingerprinting logic against real study files, producing real platform and data_type values that the test asserts against. This is a genuine integration test, not a mock roundtrip.

7. Key Findings

The process of writing tests exposed three categories of insights: confirmed behaviors, discovered bugs, and architectural lessons.

7.1 Domain Model Refactor: Confirmed Working

Fingerprinting produces correct platform/data_type fields. The pool validation integration test exercises the real fingerprint_file() function against 12 golden session files. All fingerprints carry platform and data_type fields; the old domain field does not appear. Five expected platforms (Body Weight, Organ Weight, Clinical Chemistry, Hormones, Tissue Concentration) are covered. Clinical Observations files are correctly classified under the Clinical platform with no false positives from the apical platform matcher.
Platform partitioning uses the new field. _partition_by_platform() reads experimentDescription.platform as its primary signal, falling back to the old .domain field only for legacy sessions. The unit tests confirm that when both fields are present, platform takes precedence.
Session restore works with the new format. The E2E test clears all in-memory state after approval, then reloads the session from disk. Approved sections, pending files, and chemical identity all restore correctly.

7.2 Discovered Issues

Finding 1: The responsive flag defaults to False

TableRow.responsive defaults to False, meaning that _build_section_cards() — which filters to only responsive rows for non-Body-Weight platforms — produces empty section cards unless the NTP statistical pipeline explicitly sets responsive=True. Body Weight bypasses this filter (the NIEHS reference always includes Table 2 regardless of significance).

Impact: If any code path creates TableRow objects without running the full NTP stats pipeline, the resulting section cards will be empty for all platforms except Body Weight. This would surface as missing tables in the report with no error message.

Finding 2: integrated.json is written by bmdx_pipe, not the endpoint

The api_pool_integrate endpoint’s docstring says it persists to sessions/{dtxsid}/integrated.json, but the actual disk write happens inside bmdx_pipe.integrate_pool() (the Java-backed function). The endpoint only stores the result in the in-memory _integrated_pool dict. This means the in-memory cache is the primary artifact; the disk file is a side effect of the external dependency.

Impact: If integrate_pool() changes its file-writing behavior, the process-integrated endpoint would silently lose disk persistence. Tests that mock integrate_pool must check the in-memory dict, not the disk file.

Finding 3: Golden test data lacks Hematology

The DTXSID50469320 test session has Clinical Chemistry files (clin_chem_truth_*.txt) but no Hematology files. The initial test assumed all 6 apical platforms would be present; in reality, the golden session covers only 5 (plus Clinical Observations). This reflects the actual study design for this chemical — not all studies measure all endpoints.

Impact: Tests must assert against the actual content of golden fixtures, not against an idealized schema. The fix was to remove Hematology from the expected platforms set.

7.3 Architectural Lessons

Module-level SESSIONS_DIR requires multi-site patching

Python’s from X import Y creates a new name binding in the importing module. Patching X.Y doesn’t affect modules that already imported Y by name. In this codebase, SESSIONS_DIR is imported by 6 modules; the conftest must patch all 6 to achieve true isolation. This is a common pitfall in Python testing that cost 30+ minutes of debugging during test development.

Hash functions are the cache’s correctness boundary

The per-section cache system (_cache_ntp_*.json, _cache_sections_*.json, etc.) relies on hash functions to detect when cached results are stale. The unit tests verify that these hashes are deterministic (same input → same hash) and sensitive (different inputs → different hashes). An unsound hash function would serve stale results silently — one of the hardest bugs to diagnose in production.

Prioritized Roadmap

  3. Clinical observations integration (MEDIUM) — The golden data includes clinical_obs_truth_*.csv files. Add a process-integrated test that verifies incidence tables appear in the output.
  4. Genomics fixture data (LOW) — Add a golden .bm2 file with gene expression data to test the genomics extraction and interpretation pipeline.

9. Conclusion

This test suite transforms a system that could only be validated through 30–60 minute manual browser sessions into one that can be verified in 2.3 seconds. The 125 tests span four layers of the architecture:

  • 92 unit tests validate pure functions (serialization, hashing, slug generation, tree operations, platform partitioning) with no I/O or mocking
  • 32 integration tests exercise the FastAPI endpoints through a TestClient with mocked external dependencies, verifying request/response contracts, disk persistence, and state management
  • 1 end-to-end test walks the complete pipeline (session load → validate → integrate → process → approve → restore), covering all 9 points of the domain model refactor validation checklist

The most significant result is the confirmed correctness of the domain model refactor’s fingerprinting layer. The pool validation integration test exercises the real fingerprint_file() function against 12 real study files and confirms that every fingerprint carries the new platform and data_type fields. This was the highest-risk untested assertion — the entire downstream pipeline (partitioning, section card generation, table numbering, narrative grouping) depends on these fields being correct.

Three bugs were discovered during test development, none critical but all instructive: a default responsive=False that silently empties section cards, a docstring/behavior mismatch on disk persistence, and a golden data assumption that didn’t match reality. These are exactly the kinds of issues that manual testing misses because they produce empty results rather than error messages — the UI renders nothing, and the tester has to notice the absence.

The test suite is a foundation, not a finish line. The NTP statistics pipeline, export rendering, and genomics interpretation remain untested against known-answer datasets. But the infrastructure is now in place: state isolation, mock architecture, golden fixtures, and pytest markers that let developers run the full suite in 2 seconds or target a specific layer in under 1 second. Future tests are incremental additions to a working framework, not a cold start.
