## Assistant (claude-opus-4-6)



Semantic Scholar's API rate limits work on a sliding window — when you exceed the allowed requests per time period, it responds with HTTP 429 and a `Retry-After` header (the 10s waits you see in the log). The smoke test we ran earlier (circadian crawl, 65 API calls) consumed part of that window. So the full crawl started with a partially depleted quota.

Once enough time passes (usually a few minutes), the oldest requests age out of the window and the crawler gets its full rate back. The `[rate limited, waiting 10s]` messages will become infrequent — you'll mostly see them at burst points like when expansion fetches many references in quick succession.
