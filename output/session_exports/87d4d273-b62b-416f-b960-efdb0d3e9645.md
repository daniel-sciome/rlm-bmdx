## Sprint 4: Interpretation Engine

### 9. `interpret.py` — Benchmark CSV → grounded analysis (NEW)

**This has no bmdx equivalent.** This is where the system becomes an expert.

**Input:** User provides a CSV with columns like `model, benchmark, score, metric, conditions`

**Process:**
1. Parse CSV, normalize model/benchmark names using `normalize.py`
2. For each (model, benchmark) pair, query the DB:
   - SOTA history for that benchmark (top 20 scores over time)
   - Techniques used by that model
   - Papers discussing that model on that benchmark
   - Model lineage (what is it derived from?)
3. Build a structured context block with all KB facts
4. Send to LLM with interpretation prompt: "Given these user results and this literature context, generate a grounded analysis"

**Output:** Interpretation report covering:
- Performance ranking vs known SOTA (with paper citations)
- Temporal trend analysis (how does this compare to SOTA trajectory?)
- Technique attribution (what likely explains the results?)
- Gap analysis (which important benchmarks are missing?)
- Methodological caveats (different eval conditions, etc.)

The LLM acts as a **synthesis engine** — all facts come from the KB, the LLM reasons over them and generates narrative. This is RAG applied to the literature-derived knowledge base.

## Your reading path to understanding Expert Models

### Level 0: The problem language models solve

A language model predicts the next word. Given "The cat sat on the", it assigns probabilities to every possible next word ("mat" high, "refrigerator" low). That's it. Everything else — chatbots, code generation, reasoning — emerges from training this simple objective on enormous amounts of text.

The model itself is a neural network: layers of numbers (parameters/weights) that transform an input through many stages until a prediction comes out. More parameters = more capacity to learn patterns. This is the key tension that expert models solve.

---

### Level 1: The Transformer — read this first

**"Attention is All You Need" (2017)** — 165,837 citations in your DB

This is the architecture underneath every modern LLM. The core idea: instead of reading text sequentially (word by word), the model can look at all words simultaneously and learn *which words to pay attention to* for each prediction. The "attention mechanism" is literally a learned relevance score between every pair of words.

Before this, models used recurrent networks (process one word at a time, slow). The Transformer processes all words in parallel, which made it possible to train on vastly more data.

**What to focus on:** Don't worry about the math. Understand that a Transformer is a stack of layers, each layer has "attention heads" (multiple independent attention patterns), and the output of one layer feeds into the next.

---

### Level 2: Scaling up — the brute force era

Three papers in your DB tell this story:

1. **"BERT" (2019)** — 110,050 citations. Showed that pre-training a Transformer on lots of text, *then* fine-tuning it for specific tasks, beats everything. 340M parameters.

2. **"Language Models are Few-Shot Learners" (2020)** — GPT-3. 54,082 citations. Showed that if you make the model big enough (175B parameters), it can do tasks it was never explicitly trained for — just from a few examples in the prompt. This is "emergent ability."

3. **"Scaling Laws for Neural Language Models" (2020)** — 7,046 citations. Kaplan et al. discovered that model performance improves *predictably* as you increase parameters, data, and compute. A smooth power law — double the parameters, get a predictable improvement.

**The scaling problem:** GPT-3 has 175B parameters. Every single token you process flows through *all 175 billion* of them. The compute cost scales linearly with parameter count. To get a 10x better model, you need 10x the compute *per token*. This gets expensive fast.

---

### Level 3: The Mixture of Experts idea

This is where it gets interesting. Your DB has a survey paper that frames it well:

**"A Survey on Mixture of Experts in Large Language Models" (2024)** — 224 citations

> *"The mixture of experts (MoE) has emerged as an effective method for substantially scaling up model capacity with minimal computation overhead."*

Here's the core insight, in plain terms:

**The problem:** A 175B parameter model uses all 175B parameters for every token. But does the word "cat" really need the same computation as a complex math equation? Probably not.

**The solution:** Instead of one giant feed-forward layer, split it into multiple smaller **expert** sub-networks (say, 8 of them). Add a tiny **router** network that looks at each incoming token and decides: "This token should be processed by Expert 2 and Expert 5." The other 6 experts sit idle for this token.

```
Standard Transformer layer:
  Token → [Attention] → [One big FFN with all parameters] → Output

MoE Transformer layer:
  Token → [Attention] → [Router: "pick 2 of 8"] → [Expert 2 + Expert 5] → Output
```

**Why this is powerful:** You can have 8x the total parameters (8 experts), but each token only activates 2 of them, so the compute cost per token stays roughly the same as a model with 1/4 the total parameters. You get the *knowledge capacity* of a huge model with the *compute cost* of a small one.

The key terms:
- **Expert** = a sub-network (usually a feed-forward block)
- **Router** (or **gating network**) = the small network that decides which experts handle each token
- **Sparse** = only a few experts activate per token (vs "dense" where everything activates)
- **Top-K routing** = router picks the K highest-scoring experts (usually K=2)

---

### Level 4: See it concretely — Mixtral

**"Mixtral of Experts" (2024)** — 1,648 citations in your DB

This is the clearest real-world example. From the abstract:

> *"Mixtral has the same architecture as Mistral 7B, with the difference that each layer is composed of 8 feedforward blocks (i.e. experts). For every token, at each layer, a router network selects two experts to process the current state."*

The numbers tell the story:
- **Total parameters:** 47B (8 experts x ~7B each, roughly)
- **Active parameters per token:** ~13B (only 2 of 8 experts fire)
- **Performance:** Matches or beats LLaMA 2 70B (a dense model with 70B *all-active* parameters)

So Mixtral achieves 70B-class performance at 13B-class compute cost. That's the MoE value proposition.

---

### Level 5: Advanced MoE — DeepSeek

**"DeepSeek-V2: A Strong, Economical, and Efficient Mixture-of-Experts Language Model" (2024)** — 1,000 citations

DeepSeek pushes the idea further:
- **236B total parameters, only 21B activated per token**
- Introduces **fine-grained experts** — instead of 8 big experts, use many smaller ones (more specialization)
- Novel attention mechanism (Multi-head Latent Attention) to reduce memory

Your DB also has the scaling law paper for this:

**"Scaling Laws for Fine-Grained Mixture of Experts" (2024)** — 123 citations

This shows that the "granularity" of experts matters — more, smaller experts can be better than fewer, larger ones, and this relationship follows predictable scaling laws (just like the original Kaplan scaling laws, but with an extra variable).

---

### Level 6: Why experts specialize

A natural question: do different experts actually learn different things, or is the split arbitrary?

Research from papers in your DB (like OpenMoE, 161 citations) shows that experts do develop soft specializations — some handle syntax-heavy tokens, others handle domain knowledge, others handle rare words. But the specialization isn't clean-cut; it's more like overlapping tendencies than strict division of labor.

Key challenges the literature discusses:
- **Load balancing:** If all tokens route to the same 2 experts, the other 6 are wasted. Models add a "load balancing loss" to encourage even usage.
- **Expert pruning:** Your DB has several papers ("Not All Experts are Equal", 80 citations) showing you can remove some experts post-training with minimal quality loss — meaning some experts are more important than others.
- **Routing instability:** Early in training, the router hasn't learned good routing, so expert specialization can collapse.
